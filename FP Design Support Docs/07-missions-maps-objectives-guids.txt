--------------------------------------------------------------------------------
HOW TO READ THIS GUIDE (NEW DESIGNERS)
--------------------------------------------------------------------------------

Think of a flashpoint as a TV season and each contract as one episode filmed on a
real location (the map):

  - The **poster / milestones** decide WHEN the episode airs (not covered deeply here).
  - The **contract JSON** is the script + shot list for that episode.
  - The **MapID** is the physical location (desert base, jungle river, urban grid).
  - The **encounter layer** is the pre-built set on that location — where actors
    (’Mechs) stand, where the objectives are marked on the ground.
  - **GUIDs** are sticky notes that tie the script lines to the same prop on set.

If you only change dialogue but pick the wrong location or episode type, the episode
will not film — the game refuses to load the mission. This guide aims to support you in
preventing such mismatchs.

================================================================================
PART B — MENTAL MODEL: HOW A TACTICAL MISSION IS BUILT
================================================================================

IN PLAIN ENGLISH: Part B is the “how filming works” chapter — what loads, in what
order, and which file is responsible for what. Read this once before editing JSON.

--------------------------------------------------------------------------------
B.1 From flashpoint milestone to contract load
--------------------------------------------------------------------------------

In plain English: the strategy layer (Leopard / star map) only schedules the mission;
the contract file is what actually builds the battlefield when you press Deploy.

  1. Player accepts flashpoint or a milestone fires Flashpoint_AddContract.
  2. Milestone passes contract id (e.g. c_fp_rTTJ_1_SimpleBattle) — must match
     "ID" inside the contract JSON character-for-character.
  3. Milestone Action Data often supplies Map Name and EncounterLayerGUID
     (additionalValues on Flashpoint_AddContract — 01 §3.2).
  4. Game loads ContractOverride JSON + resolves MapID + encounter layer.
  5. Tactical scene runs: chunks spawn units, objectives track progress.

The contract file does NOT load by itself; mod.json must list ContractOverride
with the correct Path (01 §3.3, 02 pathway).

--------------------------------------------------------------------------------
B.2 The contract JSON as the mission “recipe”
--------------------------------------------------------------------------------

One contract file = one drop. It has two layers:

  TOP (briefing layer) — ID, contractName, descriptions, difficulty, mapMood,
    fog, tonnage limits, contractObjectiveList, objectiveList, salvage, FP flags.

  BOTTOM (battlefield layer) — chunkList, spawners, lance overrides, cameras,
    dialogueList, buildings, artillery hooks.

Flashpoint contracts almost always use:
  "contractDisplayStyle" : "BaseFlashpoint"
  "excludedFromProceduralGeneration" : true
  "scope" : "MID_CAMPAIGN"   (or similar FP scope — see R10)

--------------------------------------------------------------------------------
B.3 MapID vs encounter layer vs contract content
--------------------------------------------------------------------------------

In plain English: three separate choices must line up — wrong combination = load error
or objectives that never complete.

  MapID — Which Unity terrain/scene file loads (e.g. mapGeneral_riverDelta_vLow).
          Lives in map data; listed in map-names.csv.

  Encounter layer — Pre-authored layout ON that map for a contract TYPE: spawn
          points, boundaries, default objectives wired to the terrain. Identified
          by EncounterLayerGUID (often MapID.uuid). Listed in EncounterLayer.csv
          filtered by ContractTypeID / encGeneral_* name.

  Contract JSON — Selects contract TYPE (implicitly via cloned template), toggles
          optional chunks, overrides lances/pilots/text, and must keep GUIDs in
          sync with the encounter layer you use.

Rule (R10): A SimpleBattle JSON cannot become a Defend Base by JSON hack alone.
You must start from a contract type that already has the right encounter layers
on your chosen map.

--------------------------------------------------------------------------------
B.4 mapMood, fog, and presentation (not the terrain file)
--------------------------------------------------------------------------------

  mapMood — Lighting/sky preset (e.g. JungleSunsetFoggy). Must exist in game data;
            typo = load failure.
  startingFogOfWarVisibility — Surveyed, Unsurveyed, etc.

These change LOOK and initial fog; they do not change which MapID loads.

--------------------------------------------------------------------------------
B.5 Chunks, chunkList, and mission phases
--------------------------------------------------------------------------------

Chunks are timed segments of the mission:
  - When waves spawn.
  - When bonus objectives activate.
  - When turrets/APCs/dropships appear.

Each chunk references an EncounterObjectGuid. enableChunkFromContract true/false
turns optional pieces on (R10: “optional chunks” on FP Reference sheet).

Fire missions and defend-base contracts have many more chunks than a simple ambush.

--------------------------------------------------------------------------------
B.6 Spawners, lances, and optional/bonus chunks
--------------------------------------------------------------------------------

Below chunkList, contracts reference LanceDefs, MechDefs, pilot ids, and
unitSpawnPointOverrideList (heraldry, custom names — 05).

targetTeam → lanceOverrideList is where opfor composition is customized without
editing the Unity map.

--------------------------------------------------------------------------------
B.7 Cameras, dialogue hooks, and mission buckets (vs strategy talks)
--------------------------------------------------------------------------------

  dialogueList in contract — Mission-start/success/failure lines on the tactical
    map (often with cameraFocusGuid pointing at spawner GUIDs).

  overrideAutoCompleteDialogueId / dialog buckets — Tactical barks during combat
    (DialogBucketDef), NOT simGameConversations .bytes (04b).

Do not confuse contract dialogueList with Leopard strategy conversations.


================================================================================
PART C — GUIDs: WHAT THEY ARE AND WHY THEY MATTER
================================================================================

IN PLAIN ENGLISH: GUIDs are matching labels inside ONE contract file. They are not
mod-wide IDs. If two blocks use the same label, the game treats them as the same
objective or spawner; change one without the other and the mission breaks.

--------------------------------------------------------------------------------
C.1 Plain-English definition (align with 01 §1.4)
--------------------------------------------------------------------------------

A GUID is a long hexadecimal label (e.g. d111d351-a666-4852-8e4b-a300d0cccc2d)
used INSIDE ONE contract file so the game knows two JSON blocks refer to the
same spawn point, objective, chunk, or camera target.

GUIDs are not mod-wide ids. They are per-contract wiring labels copied from a
parent encounter when you clone a mission.

--------------------------------------------------------------------------------
C.2 Where the same GUID must appear (pairing rules)
--------------------------------------------------------------------------------

When you change a GUID, search the ENTIRE contract file and update every copy.

Typical pairings:
  contractObjectiveList[].contractObjective.EncounterObjectGuid
    ↔ objectiveList[].objective.EncounterObjectGuid (via objectiveGuids[])
  objective GUID ↔ chunk encounterObjectGuid (when chunk gates that objective)
  cameraFocusHelperList[].encounterObject.EncounterObjectGuid ↔ spawner GUID
  dialogue cameraFocusGuid ↔ spawner or lance GUID

--------------------------------------------------------------------------------
C.3 contractObjectiveList ↔ objectiveList ↔ chunks ↔ spawners
--------------------------------------------------------------------------------

  contractObjectiveList — “Headline” objectives shown in UI; groups sub-objectives
                          via objectiveGuids[].
  objectiveList — Individual win/loss steps the player ticks off.
  chunkList — Enables spawns and phases tied to encounter objects.
  Spawners / lances — Physical units on the map; referenced by GUID in cameras
                      and sometimes objectives (“destroy this lance”).

If contractObjectiveList points at a GUID that does not exist in objectiveList,
the mission may hang at deployment or never award completion.

--------------------------------------------------------------------------------
C.4 Worked example: tracing one objective through a contract file
--------------------------------------------------------------------------------

Source: fps_TheBigDealBEXCE\contracts\RTTJ\c_fp_rTTJ_1_SimpleBattle.json

  contractObjectiveList (primary):
    EncounterObjectGuid: 73275787-720a-4c33-9f20-953b1bbf48bd
    objectiveGuids: [ d111d351-..., 0e2a07c1-... ]
    title: "Reach the NAIS Archaeological Expedition Camp"

  objectiveList entry 1:
    EncounterObjectGuid: d111d351-a666-4852-8e4b-a300d0cccc2d
    title: "Destroy the First Guard Lance"

  objectiveList entry 2:
    EncounterObjectGuid: 0e2a07c1-e22a-47a7-a353-3e93ebcd7b47
    title: "Destroy the Second Guard Lance"

  chunkList bonus:
    Chunk_Ambush (Bonus) → EncounterObjectGuid: 181d617b-02bc-4fd3-be25-d6d5cffb78de

  cameraFocusHelperList:
    Spawner_PlayerLance → 76b654a6-4f2c-4a6f-86e6-d4cf868335fe

Lesson: Clone this file → replace ALL GUIDs only if you also replace the encounter
layer with one you built. Safer: keep GUIDs, change lance defs and text only.

--------------------------------------------------------------------------------
C.5 Common GUID failure modes
--------------------------------------------------------------------------------

  - Changed one GUID in objectiveList but not in contractObjectiveList.
  - Copied chunk from another contract without copying matching objectives.
  - cameraFocusGuid points at deleted spawner after editing lances.
  - OnSuccessDialogueGUID set to a GUID that does not exist in dialogueList.
  - Merged two contracts’ lower halves → duplicate EncounterObjectGuid collisions.

--------------------------------------------------------------------------------
C.6 Safe edit and search habits before each test launch
--------------------------------------------------------------------------------

  1. Validate JSON (01 §1.3).
  2. Ctrl+F each EncounterObjectGuid you touched; count matches.
  3. Confirm milestone contract id = file "ID".
  4. One change → save → test single mission.
  5. Clear ModTek cache after large contract edits (01 §1.7).

PART D — MAPS (EXISTING MapIDs ONLY FOR FP AUTHORS)
================================================================================

IN PLAIN ENGLISH: You do not build new terrain for typical flashpoints. You pick an
existing MapID (a catalogue number for a finished map). Wrong number = wrong set or
a load error. Curated lists below are shortcuts; always confirm IncludeInBuild = 1
in the CSV files.

--------------------------------------------------------------------------------
D.1 What a MapID is and where it lives on disk
--------------------------------------------------------------------------------

MapID is the string the game uses to load a Unity scene, e.g.:
  mapGeneral_riverDelta_vLow

Vanilla map metadata lives under:
  BATTLETECH\BattleTech_Data\StreamingAssets\data\maps\
(and related encounter data). Mods do not ship new maps; FP authors reference
existing MapIDs only (D.11).

Milestones pass Map Name into Flashpoint_AddContract additionalValues; the string
must match a real MapID from map_names.csv or the mission fails at load.

--------------------------------------------------------------------------------
D.2 Friendly names vs internal MapID strings
--------------------------------------------------------------------------------

Players see contractName / briefing text YOU write. The engine uses MapID.

map-names.csv columns:
  MapID | FriendlyName | … | IncludeInBuild | …

Always author with MapID; use FriendlyName only when choosing “jungle vs desert.”

--------------------------------------------------------------------------------
D.9 Biome, TagSetID, and capability constraints
--------------------------------------------------------------------------------

  - Each MapID has TagSetID used by encounter tooling and filters.
  - Lunar maps (_bMoon, some _bMars) may restrict contract types (e.g. few rescue layers).
  - Urban maps expect building props; convoy missions may path differently.
  - Jungle maps often suit ambush / escort narratives; open desert suits fire missions.

Use EncounterLayer.csv: filter by MapID column, read ContractTypeID / encGeneral_*.

--------------------------------------------------------------------------------
D.10 Choosing a map for your narrative beat
--------------------------------------------------------------------------------

  1. Pick mission SHAPE first (Part F) — narrows legal maps.
  2. Filter map_names.csv by biome / IncludeInBuild = 1.
  3. Open EncounterLayer.csv — confirm ≥1 layer for your shape on that MapID.
  4. If multiple EncounterLayerGUIDs exist, preview in game or clone HBS contract
     that already uses that GUID in milestone additionalValues.
  5. Match mapMood to fiction (night assault → darker mood).

Author note: If anyone knows of an image repository that shows the maps for reference
please shout out.


================================================================================
PART E — ENCOUNTER LAYERS AND MAP–ENCOUNTER PAIRING
================================================================================

IN PLAIN ENGLISH: The encounter layer is the furniture layout in a furnished apartment.
The contract chooses which apartment (MapID) and which layout variant (GUID). You can
redecorate (swap lances, toggle optional chunks) but not turn a studio into a stadium
without a different layer type.

--------------------------------------------------------------------------------
E.1 What an EncounterLayer is
--------------------------------------------------------------------------------

An encounter layer is map-specific data that defines:
  - Player and enemy spawn locations.
  - Map boundary size.
  - Default objective anchors and patrol routes.
  - Which optional chunks the terrain supports.

The contract JSON selects and configures a layer; it does not draw the terrain.

--------------------------------------------------------------------------------
E.2 EncounterLayerID, MapID, and EncounterLayerGUID
--------------------------------------------------------------------------------

From EncounterLayer.csv (typical columns):
  EncounterLayerID | MapID | Name (encGeneral_Battle) | EncounterLayerGUID | …

EncounterLayerGUID is what milestones often pass in Flashpoint_AddContract
additionalValues when not using the default layer for that map+type.

Format example:
  mapGeneral_riverDelta_vLow.a92bff88-a599-47bf-95be-9a746e0f6507

Same map can have multiple GUIDs = different spawn layouts on identical terrain.

--------------------------------------------------------------------------------
E.3 ContractTypeID and encGeneral_* families
--------------------------------------------------------------------------------

encGeneral_Battle, encGeneral_Defend, etc. tie to contract template families
(simplebattle, defendbase, firemission, …). ContractTypeID numeric column in csv
maps to enumeration in game data.

You do not pick encGeneral_* directly in beginner authoring — you clone a contract
that already targets the right family for your map.

--------------------------------------------------------------------------------
E.4 One map, many encounter layers (layout variants)
--------------------------------------------------------------------------------

Example (river delta): two encGeneral_Battle layers with different GUIDs — different
enemy spawn quadrants. Same biome, different tactical feel.

When cloning RTTJ or Big Deal contracts, note which EncounterLayerGUID the
milestone passes; changing map without changing GUID can place spawns in void.

--------------------------------------------------------------------------------
E.5 When the contract JSON overrides the encounter layer
--------------------------------------------------------------------------------

Contract JSON overrides:
  - Optional chunks on/off.
  - Lance compositions, pilots, heraldry.
  - Briefing text, difficulty display, tonnage limits.
  - Some objective titles (wording) tied to existing GUIDs.

Contract JSON does NOT override:
  - Fundamental contract type (without matching encounter layer).
  - Terrain mesh or building placement (without new map mod).

--------------------------------------------------------------------------------
E.6 Mission Control: what it changes at runtime (spawn, bounds, objectives)
--------------------------------------------------------------------------------

Mission Control (MC) hooks contract generation at runtime:
  - Extra lances, adjusted spawn points, map boundary scale.
  - Custom contract types (SoloDuel, DuoDuel, Blackout — see MC ContractType.json).
  - Reuse of story maps with procedural tweaks.

MC does not replace the need for a valid base contract + MapID. See Part J.


================================================================================
PART F — MISSION TYPES AND CONTRACT SHAPES
================================================================================

IN PLAIN ENGLISH: “Mission type” is not one dropdown in the contract. It is the
combination of (1) which stock template you cloned, (2) which map + encounter layer
you paired it with, and (3) which optional chunks you turned on. Part F sorts shapes
by where they come from (base game, fire missions, OM, MC).

--------------------------------------------------------------------------------
F.0 Mission shape taxonomy (how this part is organized)
--------------------------------------------------------------------------------

Three layers of “type” (do not confuse them):

  1. Contract template family — simplebattle, firemission, captureescort, …
     (stock .txt [Template] or cloned JSON structure). Drives encounter layer pool.

  2. encounterPlayStyle — Usually SinglePlayer for FP; slot count and drop rules.

  3. Objective wording + chunks — What the player reads and which optional waves fire.

Part F.1 = vanilla/BEXT base shapes | F.2 = fire missions | F.3 = Opportunity Missions
| F.4 = MC and example mods.

--------------------------------------------------------------------------------
F.1 — BASE GAME MISSION SHAPES
--------------------------------------------------------------------------------

Stock contract template families (contracts.csv Type column):

  ambushconvoy      — Convoy ambush; optional ambush-by-kills columns
  assassinate       — Kill target; escape variants
  capturebase       — Capture base; defenders / turrets flags
  captureescort     — Escort + capture hybrid
  defendbase        — Hold base; APC / third wave / turret options
  destroybase       — Demolition / destroy structures
  rescue            — Rescue units from map
  simplebattle      — Standard fight; most FP battles start here
  firemission       — Target acquisition / artillery (Part F.2)
  attackdefend      — Attack-defend hybrid
  threewaybattle    — Three factions on map

Mission Control adds: SoloDuel, DuoDuel, Blackout (enumeration IDs 10000+).

--------------------------------------------------------------------------------
F.1.1 encounterPlayStyle and drop rules (SinglePlayer, slots, tonnage)
--------------------------------------------------------------------------------

Typical FP row:
  "encounterPlayStyle" : "SinglePlayer"
  "maxNumberOfPlayerUnits" : 4
  "mechMaxTonnages" : [ 75, 75, 75, 75 ]

Rules:
  - Array length must match maxNumberOfPlayerUnits.
  - -1 often means “no limit” for that slot (verify in cloned template).
  - lanceMinTonnage / lanceMaxTonnage cap total drop weight (fire missions use heavy
    lance mins — see c_fp_allianceSteiner_a1 in R10).
  - Forced ally slots count toward maxNumberOfPlayerUnits (R10 warning).

Field detail: 01 §3.3.

--------------------------------------------------------------------------------
F.1.2 Standard battle / destroy enemy lance
--------------------------------------------------------------------------------

Player experience: Deploy, destroy primary lance(s), optional bonus ambush chunk.

Clone from: simplebattle stock or c_fp_rTTJ_1_SimpleBattle.json.

Flexibility:
  + Toggle bonus lance chunk (Chunk_Ambush).
  + Change opfor lances and heraldry (05).
  + Multiple destroy objectives in objectiveList.

Limits:
  - Cannot add “capture base” win condition without capturebase template/layer.
  - Objective GUIDs must stay paired (Part C).

--------------------------------------------------------------------------------
F.1.3 Escort, capture escort, and moving objective targets
--------------------------------------------------------------------------------

Player experience: Protect unit(s) moving across map; may combine with capture.

Clone from:
  captureescort stock contracts
  c_fp_rTTJ_2_captureEscort.json
  c_fp_theBigDeal_5A_CaptureEscort.json

Flexibility:
  + Escort APC chunks (defendbase/captureescort sheet columns).
  + Reinforcement waves via chunk toggles.

Limits:
  - Escort path is baked into encounter layer.
  - If escort unit dies in script, mission fails — test difficulty carefully.

--------------------------------------------------------------------------------
F.1.4 Destroy base, demolition, and structure objectives
--------------------------------------------------------------------------------

Player experience: Destroy structures / base buildings; may include turrets.

Clone from:
  destroybase stock
  c_fp_theBigDeal_1_DestroyBase.json
  capturebase variants when narrative is “take facility”

Flexibility:
  + Structure list in lower contract JSON (buildingList).
  + Optional defender lance chunks.

Limits:
  - Map must include destroyable structures in that encounter layer.
  - Urban maps (D.6) behave differently for destructible buildings (UW DLC).

--------------------------------------------------------------------------------
F.1.5 Three-way battles and multi-faction contracts
--------------------------------------------------------------------------------

Player experience: Three factions fight; player may ally with one.

Clone from:
  threewaybattle stock
  c_fp_theBigDeal_4A_3wayBattle.json
  c_special_sAV_4_ThreeWayBattle.json (Big Deal / MC packs)

Flexibility:
  + Faction hostility via contract and lance teams.
  + Complex briefing still uses standard objective GUID rules.

Limits:
  - Fewer maps support threeway layers — check EncounterLayer.csv first.
  - AI alliance behaviour is finicky; playtest all three sides attacking player.

--------------------------------------------------------------------------------
F.1.6 Base defense, turrets, APCs, and bonus chunks
--------------------------------------------------------------------------------

Player experience: Hold point; waves; sometimes APC powers turrets.

Clone from: defendbase stock; Contracts.csv columns EscortAPCs, 3rd Wave, NeutralTurrets.

Flexibility:
  + Turn optional chunks on/off (R10 spreadsheet lists which stock contracts enable what).
  + Turret and APC chunks named in chunkList (e.g. Chunk_EscortAPC_DefensiveTurrets).

Limits:
  - Wrong map + defend layer = missing APC path.
  - Cannot add brand-new turret type without TurretDef mod content.

--------------------------------------------------------------------------------
F.1.7 Vehicle, convoy, and mixed-unit missions
--------------------------------------------------------------------------------

Player experience: Convoys, vehicles, mixed lances.

Clone from: ambushconvoy, captureescort, some defendbase; vehicle LanceDefs in contract.

Flexibility:
  + VehicleDef / convoy lances in lower JSON.
  + Custom unit names on spawn points (05).

Limits:
  - Vehicle pathing tied to map layer.
  - Requires vehicle defs in mod or vanilla dependency.

F.2 — FIRE MISSIONS / TARGET ACQUISITION (FULL TREATMENT)

--------------------------------------------------------------------------------
F.2.1 What makes a fire mission different from a standard contract
--------------------------------------------------------------------------------

Fire missions (template FireMission) are target-acquisition set pieces:
  - Player paints targets for off-map artillery / dropship strikes.
  - Strict time pressure and phase gates (clear AA, place beacons, etc.).
  - Much larger lower JSON: artilleryObjectiveList, platforms, many chunks.

They use the same GUID discipline but more moving parts than simplebattle.

--------------------------------------------------------------------------------
F.2.2 HBS template and design-data references (Flashpoint DLC examples)
--------------------------------------------------------------------------------

Design .txt templates (parser source):
  bt-dlc-designdata-master\Flashpoint\design\contracts\flashpoint\
    allianceSteiner\c_fp_allianceSteiner_a1_fireMission.txt
    allianceMarik\c_fp_allianceMarik_*.txt
    scoundrel\c_fp_scoundrel_a1_target.txt
    unwelcomeGuests\c_fp_unwelcomeGuests_1_fireMission.txt

Tags in template:
  [Template]FireMission
  [ContractObjective] with sub-objectives for beacon / AA / timing

Shipped JSON equivalents live under StreamingAssets after HBS build; clone JSON
from DLC or BEXT packs for mod work.

--------------------------------------------------------------------------------
F.2.3 Map and encounter-layer constraints for fire missions
--------------------------------------------------------------------------------

Not every map has a firemission encounter layer. Filter EncounterLayer.csv for
firemission contract type rows on your MapID.

Prefer maps HBS used for FireMission stock contracts (open terrain, clear LOS).
Lunar rescue-style single-layout maps are poor fits.

Part H.5 matrix marks fire-mission eligibility by map family.

--------------------------------------------------------------------------------
F.2.4 Chunks, artillery, platforms, and file size expectations
--------------------------------------------------------------------------------

Expect multi-thousand-line JSON when cloning FP fire missions (e.g. c_fp_rTTJ_3_fireMission.json).

Key blocks:
  artilleryObjectiveList
  buildingList / platform references
  Many named chunks for phases (AA defense, beacon, volley)
  dialogueList tied to phase transitions

Do not truncate the lower half when cloning — missing chunk = soft-lock mid-mission.

--------------------------------------------------------------------------------
F.2.5 Why the R10 parser is not used; hand-authoring and clone workflow
--------------------------------------------------------------------------------

R10 (..\Background Docs\r10-flashpoint-authors-guide.txt) explicitly: Contract Parser does NOT handle FireMission
templates at this time.

Recommended workflow:
  1. Clone nearest HBS or BEXT firemission JSON.
  2. Rename ID / contractName / descriptions.
  3. Adjust tonnage and lance overrides only until experienced.
  4. Keep GUIDs and chunk structure intact.
  5. Change milestone MapID + EncounterLayerGUID together if switching maps.

--------------------------------------------------------------------------------
F.2.6 Worked field highlights (key JSON blocks only — detail in 01 §3.3)
--------------------------------------------------------------------------------

  Field / block              | Fire-mission notes
  ---------------------------|------------------------------------------
  [Template] / structure     | Must remain FireMission family
  artilleryObjectiveList     | Orbital / artillery strike definitions
  lanceMinTonnage            | Often high total tonnage (e.g. 320)
  chunkList                  | Phase gates; enableChunkFromContract per phase
  excludedFromProcedural...  | true for FP
  contractDisplayStyle       | BaseFlashpoint

--------------------------------------------------------------------------------
F.2.7 Common fire-mission failure modes
--------------------------------------------------------------------------------

  - Parser-generated simplebattle used where fire layer required.
  - Map swap without compatible fire encounter layer.
  - Disabled chunk that still held required objective GUID.
  - Artillery objective GUID not linked to contractObjectiveList.
  - Timeout narrative not communicated (players unaware of phase timer).

================================================================================
PART G — OBJECTIVES
================================================================================

IN PLAIN ENGLISH: Objectives are what the player reads on the left of the screen.
Under the hood they are wires to encounter objects on the map. You can rewrite the
words freely; you cannot usually change “destroy base” into “escort convoy” without
changing the whole contract template and map layer.

--------------------------------------------------------------------------------
G.1 Player-visible objectives vs behind-the-scenes wiring
--------------------------------------------------------------------------------

Players see title + description from contractObjectiveList / objectiveList.

Behind the scenes:
  - EncounterObjectGuid links UI objectives to encounter objects on the layer.
  - Chunks activate/deactivate groups of encounter objects.
  - OnSuccessResults / OnFailureResults on objectives can fire tags (often empty in FP).

--------------------------------------------------------------------------------
G.2 contractObjectiveList (contract-level glue)
--------------------------------------------------------------------------------

Each entry:
  contractObjective.EncounterObjectGuid — parent encounter object
  title / description — briefing and in-mission header text
  isPrimary — primary vs secondary display
  forPlayer — usually Player1
  objectiveGuids[] — list of child objective GUIDs from objectiveList

One parent can group multiple destroy steps (“Reach camp” + destroy lances).

--------------------------------------------------------------------------------
G.3 objectiveList (step objectives and hooks)
--------------------------------------------------------------------------------

Each entry:
  objective.EncounterObjectGuid
  title / description / isPrimary
  OnSuccessDialogueGUID / OnFailureDialogueGUID — optional; null if unused
  OnSuccessResults / OnFailureResults — strategy tags (usually [] in FP)

--------------------------------------------------------------------------------
G.4 Exhaustive ObjectiveType / equivalent values (from game data)
--------------------------------------------------------------------------------

BattleTech does not use one JSON field called ObjectiveType in contract files.
Designers need three parallel vocabularies:

A) CONTRACT TEMPLATE FAMILY (pick when you clone — drives encounter pool)
   Source: ..\Background Docs\reference-sheets\Contracts.csv (validated May 2026). Exhaustive stock list:

     ambushconvoy    — Attack convoys; optional AmbushByNumKilled chunks
     assassinate     — Kill target; optional Target Escape chunks
     capturebase     — Take a base; defender lance / turret chunk flags
     captureescort   — Escort + capture; BlockingLance / Reinforcement columns
     defendbase      — Hold base; EscortAPCs / 3rd Wave / NeutralTurrets columns
     destroybase     — Demolish structures
     rescue          — Rescue units
     simplebattle    — Standard destroy-enemy fights
     firemission     — Target acquisition / artillery (Part F.2)
     attackdefend    — Attack-defend hybrid
     threewaybattle  — Three factions

B) ENCOUNTER OBJECT SubType (engine wiring on the map — not in contract JSON text)
   HBS stores encounter geometry in Unity assets, not in StreamingAssets\*.json.
   A grep of 3,922 StreamingAssets JSON files found zero "SubType" fields there.

   Documented SubTypes from Mission Control contractTypeBuilds (Mods\MissionControl,
   May 2026) — use as the best public list of encounter-object behaviours:

     Boundary            — Map edge / play boundary
     Container           — Groups child objectives
     DefendXUnits        — Keep specific units alive
     DestroyLance        — Destroy a lance
     DestroyWholeLance   — Destroy a lance entity as a whole
     Dialogue            — In-mission dialogue trigger
     DisablePilotDeath   — Prevent pilot death flag
     EncounterBoundary   — Encounter scope boundary
     EncounterStructure  — Structure placement hook
     Lance               — Enemy/friendly lance spawn
     Normal              — Generic/normalized objective helper
     OccupyRegion        — Hold or investigate a map region (Blackout)
     Placement           — Place structure or object
     PlayerLance         — Player spawn lance
     Simple              — Simple dialogue / helper node
     SimpleLance         — Lance with default setup

   If a future game patch adds SubTypes, grep:
     Mods\MissionControl\contractTypeBuilds\**\*.jsonc

C) CONTRACT CHUNK NAMES (optional waves you toggle true/false in chunkList)
   Source: vanilla StreamingAssets\data\contracts\ (40 unique Chunk_* names found).
   Examples designers touch most often:

     Chunk_Ambush (Bonus)
     Chunk_AmbushByNumKilled (Bonus)
     Chunk_BaseDefense_Turrets (Bonus)
     Chunk_BlockingLance (Bonus)
     Chunk_DestroyLance_Defenders (Bonus)
     Chunk_EscortAPC_DefensiveTurrets (Bonus)
     Chunk_ReinforcementLance (Bonus)
     Chunk_TargetEscape (Bonus)
     Chunk_Dropship_Extraction
     Chunk_HostileAll_Lance_Primary / _Secondary  (three-way battles)

   Full list: see M.4 appendix. Contracts.csv optional-chunk columns tell you which
   stock templates support which chunks (Ambush, Target Escape, EscortAPCs, etc.).

D) OBJECTIVE TITLE TOKENS (hidden/script objectives in complex contracts)
   Some contracts use objectiveList titles like Objective_Wave1Flag — engine script
   hooks, not player-facing copy. Found 11 unique Objective_* titles in vanilla JSON
   (mostly three-way battles). Do not delete these when cloning three-way files.

E) UI-LEVEL FLAGS (what you edit in contract JSON)
     isPrimary — Main vs secondary display / fail behaviour
     forPlayer — Usually Player1
     title / description — Player-readable text (safe to rewrite)

When 01 or R10 say “objective type,” they usually mean (A) contract family, not (B).

--------------------------------------------------------------------------------
G.5 Primary, secondary, and bonus objectives
--------------------------------------------------------------------------------

  isPrimary : true — Shown as main goals; failure may fail mission.
  Bonus chunks — Often optional lances (Chunk_* (Bonus)); toggled via chunkList.
  “OPTIONAL” in title — Player guidance; engine still uses isPrimary flags correctly.

Mislabeling secondary as primary causes harsh fail states.

--------------------------------------------------------------------------------
G.6 OnSuccess / OnFailure / dialogue tied to objectives
--------------------------------------------------------------------------------

  OnSuccessDialogueGUID — Plays dialogueList entry when objective completes.
  Must reference dialogue.EncounterObjectGuid in same file.

Prefer milestone ForceEvents for story beats; keep contract objectives focused on combat.

--------------------------------------------------------------------------------
G.7 What you can design with objectives (flexibility)
--------------------------------------------------------------------------------

  + Multi-step destroy sequences (ordered or parallel GUIDs).
  + Optional ambush lance via bonus chunk.
  + Custom objective titles/descriptions for fiction.
  + Tie camera dialogue to spawner GUIDs on success/fail.
  + Mix primary destroy with reach/escort IF template supports it on that map.

--------------------------------------------------------------------------------
G.8 What objectives cannot do (hard limits)
--------------------------------------------------------------------------------

  - Cannot change fundamental win condition type without correct contract template + layer.
  - Cannot add new encounter objects without encounter layer / MC support.
  - Cannot run strategy-layer flashpoint branch logic from objective JSON (use milestones).
  - Cannot reference another contract file’s GUIDs.
  - Cannot grant flashpoint rewards reliably from OnSuccessResults if milestone also pays
    (double-pay risk — 01 §3.3).

--------------------------------------------------------------------------------
G.9 Worked field table: objective fields that most often break missions
--------------------------------------------------------------------------------

  Field                         | Break if wrong
  ------------------------------|------------------------------------------
  EncounterObjectGuid           | Typo / orphan → hang or no progress
  objectiveGuids[] in parent    | Missing child → parent never completes
  isPrimary                     | Wrong fail logic
  OnSuccessDialogueGUID         | Points at missing dialogue block
  enableChunkFromContract       | false when objective needs that spawn
  contractObjectiveList empty   | No UI goals / instant odd states

PART H — COMPATIBILITY: MAP × MISSION-SHAPE MATRIX
================================================================================

IN PLAIN ENGLISH: You cannot put every mission type on every map. The matrix counts
how many different MapIDs in each biome family have at least one encounter layer for
that mission shape. A dot (.) means zero — do not assume that shape works there.

--------------------------------------------------------------------------------
H.1 How to read the compatibility matrix
--------------------------------------------------------------------------------

Validated pivot from ..\Background Docs\reference-sheets\EncounterLayer.csv + map_names.csv (IncludeInBuild=1
maps only, May 2026). Re-run ..\Background Docs\tools\validate-encounter-matrix.ps1 after spreadsheet updates.

How numbers were produced:
  - Each row is a map “family” (biome suffix on MapID — see D.3).
  - Each column is a mission shape inferred from encounter layer Name (encGeneral_*).
  - Cell value = count of distinct MapIDs in that family with ≥1 matching layer.
  - "." = zero maps in that family for that shape.

Before shipping YOUR mission:
  1. Filter EncounterLayer.csv by your exact MapID (not just the family).
  2. Confirm a row exists for your shape (encGeneral_Battle, encGeneral_FireMission, …).
  3. Copy EncounterLayerGUID from that row into milestone additionalValues.

Story / restoration maps (H.3): layers use bespoke names (encStory_Tutorial,
EncounterLayer_StoryEncounter*, FinalShowdown) — not encGeneral_Battle. Clone the
HBS contract that already targets that MapID; do not assume procedural shapes exist.

H.5 Fire-mission-eligible maps (from validated pivot)

H.6 Chunk and spawner requirements tied to specific maps
--------------------------------------------------------------------------------

  - Defend/capture with APC: map layer must include APC path chunks (defendbase csv flags).
  - Turret chunks: map must have turret placements in encounter layer.
  - Three-way: layer must define third faction spawn zones.
  - Urban: building destruction chunks require destructible building props.

If optional chunk toggled true but layer lacks objects → empty spawn or crash.

--------------------------------------------------------------------------------
H.7 DLC-only assets on a player install without that DLC
--------------------------------------------------------------------------------

  - _uTech maps without Urban Warfare → load failure or black screen.
  - Flashpoint DLC-only story layers without FP DLC → missing encounter.
  - Prefer D.3 general pool for widest player compatibility.

State DLC requirements in mod description and DependsOn where applicable.

--------------------------------------------------------------------------------
H.8 Mod dependency when a mission needs another mod’s encounter layer
--------------------------------------------------------------------------------

  Mission Control — runtime spawn/boundary tweaks; list in DependsOn if used.
  Opportunity Missions — custom opfor defs; DependsOn if referenced.
  Custom map mods — DependsOn + MapID from that mod.

Never assume vanilla encounter layers include mod-only MapIDs.

PART J — AUTHORING TOOLS, PATHS, AND KEY RESOURCES
================================================================================

IN PLAIN ENGLISH: Part J answers “what tool do I use when?” — cloning is the default;
the parser helps for standard shapes; Mission Control tweaks things after the mission loads.

--------------------------------------------------------------------------------
J.1 Recommended authoring path (clone JSON → parser → Mission Control)
--------------------------------------------------------------------------------

For TIDs on BEXT:

  1. CLONE JSON (default) — Copy nearest working c_fp_* or stock contract; rename IDs;
     edit briefing, lances, chunks cautiously. Fastest for FP arcs.

  2. PARSER (.txt) — Use ContractParser on design .txt for non-fire shapes when you
     prefer template syntax; output to mod contracts folder. Good for bulk variants.

  3. MISSION CONTROL — After base contract works, use MC docs to tune spawns/bounds or
     adopt MC contract types; add DependsOn.

Fire missions: skip parser (F.2.5) — clone JSON only.

--------------------------------------------------------------------------------
J.2 Path A: clone vanilla or DLC contract JSON
--------------------------------------------------------------------------------

  1. Find shape in Mods\fps_TheBigDealBEXCE or StreamingAssets\data\contracts\.
  2. Copy to your mod contracts\ folder.
  3. Set unique "ID", contractName, excludedFromProceduralGeneration true.
  4. Wire milestone Flashpoint_AddContract to that ID.
  5. Pass MapID + EncounterLayerGUID in milestone additionalValues (match source).
  6. Test one mission; grep GUIDs if you edit lower half.

--------------------------------------------------------------------------------
J.3 Path B: R10 contract parser and .txt templates (limits)
--------------------------------------------------------------------------------

  Parser: BattleTech_Data\StreamingAssets\design\contracts (or stand-alone editor folder)
  Guide: ..\Background Docs\r10-flashpoint-authors-guide.txt, HBS Eck Contract Parser Guide

  Edit .bat to point at YOUR design subfolder only (do not parse all stock .txt).

  Limits:
    - No FireMission template support (F.2.5).
    - Output must still be manifest-listed as ContractOverride.
    - Validate JSON after parse (01 §1.3).

--------------------------------------------------------------------------------
J.4 Path C: Mission Control workflow for TIDs
--------------------------------------------------------------------------------

--------------------------------------------------------------------------------
J.4.1 When to open Mission Control (before / during / after JSON edit)
--------------------------------------------------------------------------------

  BEFORE — Read MC docs to see if your shape is procedural (SoloDuel) vs override JSON.
  DURING — Tune spawn/boundary after base contract loads once.
  AFTER — If mission loads but feels unfair (spawn behind player), adjust MC settings
          rather than rewriting entire encounter by hand.

--------------------------------------------------------------------------------
J.4.2 What Mission Control changes vs what stays in contract JSON
--------------------------------------------------------------------------------

  Stays in contract JSON:
    ID, briefing, objective text, lance overrides, chunk toggles, GUID wiring, mapMood.

  MC can change at runtime (per MC config):
    Lance count, spawn positions, boundary size, some objective parameters.

  MC does not replace invalid MapID or wrong contract template family.

--------------------------------------------------------------------------------
J.4.3 Mission Control + flashpoints (DependsOn, config layout)
--------------------------------------------------------------------------------

  Add MissionControl to mod.json DependsOn when using MC features.
  MC config\Contracts\ holds examples — reference, do not copy blindly into FP pack.
  Milestones still reference your c_fp_* id strings.

--------------------------------------------------------------------------------
J.5 Key resource: Mission Control (GitHub and website)
--------------------------------------------------------------------------------

--------------------------------------------------------------------------------

================================================================================
PART K — DESIGNER WORKFLOW AND CHECKLISTS
================================================================================

IN PLAIN ENGLISH: Part K is your cheat-sheet order of operations and pre-flight
checks so you do not lose an afternoon to a missing comma or wrong MapID.

--------------------------------------------------------------------------------
K.1 Order of decisions: story → map → shape → objectives → spawns
--------------------------------------------------------------------------------

  1. Story beat (03) — What must happen in fiction?
  2. Mission shape (F) — simplebattle vs firemission vs escort…
  3. MapID (D) — Biome + IncludeInBuild + EncounterLayer.csv
  4. EncounterLayerGUID — From csv or clone source milestone
  5. Contract clone — Path J.2
  6. Objectives text + GUID discipline (C, G)
  7. Opfor lances + heraldry (05)
  8. Milestone Flashpoint_AddContract + additionalValues
  9. Playtest one mission → then next beat

--------------------------------------------------------------------------------
K.2 Cloning a mission safely (what to rename and grep)
--------------------------------------------------------------------------------

  MUST rename: "ID", file name, contractName (if desired), milestone references.
  MUST grep:   every EncounterObjectGuid if you change lower-half structure.
  COPY AS-IS:  GUID set when keeping same map+layer as source.
  CHECK:       excludedFromProceduralGeneration, contractDisplayStyle BaseFlashpoint.

--------------------------------------------------------------------------------
K.3 excludedFromProceduralGeneration and FP-only contracts
--------------------------------------------------------------------------------

  "excludedFromProceduralGeneration" : true

Missing = mission appears on random career board (R10 shipping mistake). Grep all
c_fp_* before release.

--------------------------------------------------------------------------------
K.4 One-change test discipline
--------------------------------------------------------------------------------

Change one of: MapID, layer GUID, chunk toggle, lance def, objective text.
Save → validate JSON → launch → test that contract only.
Keeps failure diagnosis local (01 §3.3 habits).

--------------------------------------------------------------------------------
K.5 “Mission will not load” triage (ordered checks — pointer to 01 §3.3)
--------------------------------------------------------------------------------

Short order (full detail 01 §3.3):
  1. Mod + DependsOn enabled
  2. mod.json ContractOverride Path
  3. Valid JSON
  4. Milestone id = contract "ID"
  5. MapID, mapMood, Lance/Mech defs exist
  6. excludedFromProceduralGeneration true for FP
  7. GUID pairs intact
  8. Clear .modtek cache
  9. output_log.txt search contract ID
  10. Duplicate contract ID with another mod

--------------------------------------------------------------------------------
K.6 Handoff fields for 04 template Part 4 / contract worksheets
--------------------------------------------------------------------------------

Capture per mission in 04 Part 4:
  - Contract id (c_fp_*)
  - contractName (player title)
  - Template family / shape (F.1 table)
  - MapID
  - EncounterLayerGUID (if non-default)
  - mapMood, fog
  - maxNumberOfPlayerUnits, tonnage caps
  - Optional chunk intent (which bonus chunks on)
  - Clone source file path (for agent reference)
  - DLC requirement (UW, FP DLC, OM, MC)

IN PLAIN ENGLISH: Appendices are lookup tables. M.5 is the same data as Part H.2 but
kept here for printing. Re-validate after game or spreadsheet updates.

--------------------------------------------------------------------------------
M.1 MapID families and naming patterns
--------------------------------------------------------------------------------

  Prefix / suffix        | Meaning (plain English)
  -----------------------|------------------------------------------
  mapGeneral_*           | Standard procedural maps (use D.3 lists)
  mapStory_* / MapStory_*| Campaign set-piece maps (clone HBS — H.3)
  mapRestoration_*       | Restoration DLC battle maps
  mapArena_*             | Arena / skirmish (usually not for FP)
  mapTest_* / test_*     | Dev maps — do not ship (D.8)
  *_uTech                | Urban Warfare city maps (D.6, UW DLC)
  *_vHigh / *_vLow       | Highland / lowland biomes
  *_vJung                | Jungle
  *_aDes                 | Desert
  *_aBad / *_aWst        | Badlands / western
  *_bMoon / *_bMars      | Moon / Mars
  *_iGlc / *_iTnd        | Glacier / tundra (Heavy Metal terrain)

IncludeInBuild = 1 in map_names.csv required for release-ready FP missions.

--------------------------------------------------------------------------------
M.2 EncounterLayer naming patterns
--------------------------------------------------------------------------------

Common Name column values in EncounterLayer.csv (build maps):

  encGeneral_Battle (+ numbered variants)       → simplebattle family
  encGeneral_DestroyBase                        → destroybase
  encGeneral_CaptureBase / CaptureEscort        → capture / escort
  encGeneral_DefendBase                         → defendbase
  encGeneral_Rescue                             → rescue
  encGeneral_FireMission                        → firemission
  encGeneral_ThreeWayBattle                     → threewaybattle
  encGeneral_AmbushConvoy / Assassinate         → ambush / assassinate
  encGeneral_AttackDefend                       → attackdefend
  encStory_* / EncounterLayer_Story*            → story-only bespoke layers

EncounterLayerGUID format: MapID + "." + uuid (paste into milestone additionalValues).
