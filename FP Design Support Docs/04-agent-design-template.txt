0.6 Communicating outside the template fields (for the human designer)
--------------------------------------------------------------------------------
  Use a short "Designer notes" block at the top of your filled template (or a
  separate text file in the mod folder) when intent does not fit a bracket field:

    DESIGNER NOTES (free text)
    - Tone / VO direction: [e.g. weary corporate lawyer, not action hero]
    - Dependencies: [must load BT_Extended_Timeline before this FP]
    - Known engine limits: [no more than 3 missions without refit if AllowRefitTime false]
    - Art already commissioned: [title card WIP — use placeholder Icon blank]
    - Questions for agent: [list explicit yes/no decisions you want proposed]

  When handing work to an AI assistant, paste or attach:
    1) This completed template (or the Parts you finished).
    2) Designer notes (above) for anything emotional, legal, or structural.
    3) output_log.txt excerpts after a failed test (mod Name, MissingDef lines).
    4) Pointers to reference arcs: "match branching density to fp_theRaid" beats
       guessing from blank fields.

  Ask the agent to quote which Part/field each answer maps to. If the agent must
  invent ids or story facts, require it to list them as "PROPOSED — confirm" items
  before generating JSON.

PART 0 — INSTRUCTIONS FOR THE AI AGENT (READ FIRST)
================================================================================

--------------------------------------------------------------------------------
0.1 Your role
--------------------------------------------------------------------------------
  - Treat this document as the single source of design intent from the designer.
  - Do not invent new story canon unless the designer explicitly approves.
  - Prefer exact id strings (snake_case / fp_ / ms_fp_ / c_fp_ habits per Guide §3).
  - Every id you create must be traced in Flashpoint File Relationships Pathway.txt
    before the mod is considered “wired.”

--------------------------------------------------------------------------------
0.2 What you must ask the designer if missing
--------------------------------------------------------------------------------
  [ ] Mod technical Name (for mod.json), display Title, author, DependsOn list
  [ ] Difficulty intent (skulls), branch philosophy (binary / multi-path / cancel?)
  [ ] Planet tag strategy (vanilla tags only vs new tags / enum work)
  [ ] Voice of employer, who speaks in Leopard scenes, tone (grim / heroic / noir)
  [ ] How many missions, mission types, and whether maps are vanilla-only
  [ ] Reward philosophy (item pool ids vs bespoke gear — needs framework alignment)
  [ ] Calendar vs sim-day limits — hard dates (06 Part C) vs AvailableTimeout only?
  [ ] Timeline mod stack in DependsOn (Timeline, BT_Extended_Timeline, FPLore?)

--------------------------------------------------------------------------------
0.3 What you must output when the template is “complete”
--------------------------------------------------------------------------------
  - A file/folder plan: list every JSON/.bytes filename stem you will create.
  - A id registry: FlashpointDef.Id, MilestoneSetID, first milestone id, each
    contract ID, each LanceDef id, new MechDef/ChassisDef ids if any, plus
    uixTxrIcon_* / heraldrydef_* / textureLogoID / FactionDef icon ids (05 Part I).
  - Calendar registry (06): start/end timeline_* or custom tags; ForcedTimelineEvent
    FE-* stems; SimGameEventDef ids; fp_finished_* exclusion tags; DependsOn timeline mods.
  - mod.json Manifest draft (Type + Path rows only; designer confirms flags).
  - Milestone graph: for each beat, list Actions (types + target ids).
  - Conversation plan: ConverseTek graphs named + hex ids AFTER export (or TBD).
  - For first-time structure, diff your milestone + poster JSON against the closest
    vanilla fp_TEMPLATE_* / ms_fp_TEMPLATE_* pair (Part 4.0) instead of inventing beat
    order cold.

--------------------------------------------------------------------------------
0.4 Hard guardrails
--------------------------------------------------------------------------------
  - JSON must validate (Guide §1.3). GUIDs in contracts must be consistent (§1.4).
  - System_StartConversation “value” must equal the conversation graph id from
    ConverseTek export — not the .bytes filename (Guide §3.4).
  - ContractOverride internal “ID” must match milestone Flashpoint_AddContract strings.
  - LanceDef unitId / pilotId must resolve to loaded MechDef / PilotDef.

--------------------------------------------------------------------------------
0.5 Escalation
--------------------------------------------------------------------------------
  If the designer requests behaviour you cannot express with JSON + stock
  milestones alone, flag “needs DLL / framework / Mission Control” and stop —
  do not promise engine features the Guide does not document.

--------------------------------------------------------------------------------
