1. DIALOG TAG REFERENCE (designer cheat sheet)
================================================================================

AUDIENCE: Designer — tag syntax reference while drafting §2–§3.

Re-verify on: https://github.com/CWolfs/ConverseTek/wiki

--- Curly-brace replacements (runtime text) ---

  {COMMANDER.FirstName}
    What: Inserts the commander's first name at play time.
    Why: Personalises briefing dialogue without separate lines per gender/name.
    Example: "Commander {COMMANDER.FirstName}, we have a contract offer."

  {COMMANDER.LastName}
    What: Commander surname.
    Why: Formal address in House or legal briefings.
    Example: "The estate of Commander {COMMANDER.LastName} will be notified."

  {COMPANY.DaysPassed}
    What: Number of in-game days since the campaign started.
    Why: Time pressure, deadlines, "you've been on-world X days" beats.
    Example: "In the {COMPANY.DaysPassed} days since we landed, the situation has worsened."

  {TGT_SYSTEM.*} / {TGT_FLASHPOINT.*}
    What: Reflects current star system or active flashpoint context (verify token in CT).
    Why: Location-aware lines without hard-coding planet names in every branch.
    Example: Check ConverseTek token list for your version before shipping.

--- Square-bracket tooltips (DataManager) ---

  [[DM.BaseDescriptionDefs[id],Label]]
    What: Hover tooltip pulling lore text from a description def.
    Why: Teaches players faction/history without cluttering the main line.
    Example: [[DM.BaseDescriptionDefs[LoreThomasCalderon],Protector Calderon]]

  [[DM.Factions[faction_Id],Label]]
    What: Tooltip for a faction name.
    Why: Employer or enemy nation context on first mention.
    Example: [[DM.Factions[faction_Davion],House Davion]]

  [[DM.CastDefs[castDef_Id],Label]]
    What: Tooltip for a character.
    Why: Introduces a named NPC the player may not know yet.
    Example: [[DM.CastDefs[castDef_DariusDefault],Darius]]

--- Flashpoint decision styling ---

  <color=#85DBF6FF>[Flashpoint Decision: …]</color>
    What: Marks a player choice that steers the flashpoint arc (colour in CT).
    Why: Players recognise story forks; pairs with ScopeTag actions.
    Example: "<color=#85DBF6FF>[Flashpoint Decision: Accept the contract]</color>"

  Apply Flashpoint ScopeTag (ConverseTek action)
    What: Writes fp_choice_* tags to flashpoint scope when the player picks a line.
    Why: Milestone Required Tags read those tags on later beats.
    Example: Action on response node → fp_choice_a

--- Conditions and actions (verify in your CT + ExtendedConversations build) ---

  Conditions (examples): commander_youth_* , tag fp_choice_a , company tags
    What: Hides or shows a response based on history or prior choices.
    Why: Branch memory and character backstory gates.

  Actions (examples): Apply Flashpoint ScopeTag , Play BattleTech Audio Event , Set BattleTech Fade
    What: Fires when the node/response is taken — audio, fades, tag writes.
    Why: Pacing and mechanical consequences of dialogue.

--- Camera / room / presentation (strategy layer — not tactical map cameras) ---

  Leopard dropship ROOM (set in milestone ms_*.json, not in 04b line text):
    System_ForceDropshipRoom "value" examples: Contract , Argo , Barracks , Engineering , Medbay
    Why: Which interior the player sees before/after the conversation.

  ConverseTek node actions (audio / fade — pick in CT Actions tab):
    Play BattleTech Audio Event — stinger, ambient, UI sound
    Set BattleTech Fade — fade out/in between beats

  Holoscreen background art (if custom):
    Texture key habit: uixTxrConv_<scene>.png (01 Reference Ch.5; ConversationTexture addendum)
    Note here in §2 per node: Holoscreen plate id or "vanilla default"

  Speaker framing:
    Cast id (castDef_*) vs Speaker id — use one per node; CT clears Cast if Speaker is set

  NOT in this doc: contract cameraFocusGuid / tactical cameras (01 Reference Ch.6 — mission JSON only).

Limits (game + CT FAQ):
  - Max 10 player responses on one prompt node (11+ breaks in-game).
  - Empty response nodes auto-follow — use conditions to block re-use.
  - "Only Once" vs "Always Show" — Always Show overrides Only Once.
  - Structure: empty root → non-empty node → non-empty response (minimum viable graph).

5. CONVERSATION JSON (ConverseTek-importable deliverable)
================================================================================

AUDIENCE: Agent builds deliverable; Designer imports in ConverseTek and saves .bytes.

What this file is
  A .json file on disk containing the serialized game Conversation object — the
  same shape ConverseTek produces when you Export a conversation to JSON from the
  editor (ConversationService.ImportConversation reads the file with JsonConvert
  into isogame.Conversation, then saves .bytes).

How the agent should build it (recommended)
  1. In ConverseTek, export JSON from a vanilla or mod conversation closest to your
     scene (similar cast, branch count, Flashpoint actions).
  2. Save as conv_fp_<story>_<purpose>.json in the mod or a handoff folder.
  3. Patch dialogue text, speaker ids, conditions, and actions from §2–§4.
  4. Preserve idRef / node linking conventions from the template export.
  5. Deliver the .json to the designer with a short change log (§10.3).

If building JSON without CT open
  - You MUST match an exported Conversation JSON sample from the designer's CT version.
  - Do not invent property names; missing or wrong types cause silent import failure.
  - After import, designer must open CT — if the graph is empty or broken, diff against
    a known-good export.

Pre-import checklist (agent)
  [ ] Empty root node present at conversation start
  [ ] Every player-facing branch has non-empty dialogue where required
  [ ] ≤10 responses on any single prompt node
  [ ] Flashpoint ScopeTag actions use ExtendedConversations defs if stack requires it
  [ ] Cast id vs Speaker id: only one active per node (CT clears Cast when Speaker set)
  [ ] Filename stem matches planned .bytes name in mod simGameConversations\

Designer: ConverseTek import steps
  1. Launch ConverseTek; set Working Directory (folder for scratch or mod convos).
  2. Use Import and select the agent's conv_fp_*.json (or File menu equivalent).
  3. CT converts JSON → .bytes in the working directory; open the conversation in the editor.
  4. Validate branches, test tags, regenerate node ids if you merged a cloned graph.
  5. Save; copy final .bytes to Mods\<YourPack>\simGameConversations\.
  6. Record graph hex id in §6 below and in 04 Part 6.4 + milestone JSON.

8. FOR THE DESIGNER (AFTER WORKSHEETS OR JSON HANDOFF)
================================================================================

AUDIENCE: Designer — post-handoff CT and in-game checks.

--------------------------------------------------------------------------------
8.1 What you should do in ConverseTek (not in chat)
--------------------------------------------------------------------------------
  - Import the agent's conv_fp_*.json; confirm the graph matches §2 intent.
  - Fix structure issues CT highlights (empty follow responses, link-backs).
  - Export/save .bytes; copy hex id to §6 and 04 Part 6.4.
  - Place .bytes under simGameConversations\; confirm mod.json Manifest row.
  - Patch ms_fp_* System_StartConversation "value" fields (hex only).

--------------------------------------------------------------------------------
8.2 What you should review in-game
--------------------------------------------------------------------------------
  - Conversation fires from the expected milestone beat (no silent skip).
  - Each Flashpoint Decision applies the fp_choice_* tags in §3.
  - Next milestone branch matches tags (no stale fp_choice_* from earlier arcs).
  - Cancel / TalkForCancel path works if authored.
  - output_log: no StartConversation / MissingDef errors for your mod Name.

--------------------------------------------------------------------------------
8.3 Handoff back to an AI assistant
--------------------------------------------------------------------------------
  Provide:
    - This completed 04b (§2–§3 + §6)
    - The Conversation JSON you imported (or CT export after your edits)
    - output_log fragments mentioning StartConversation, MissingDef, or your mod Name
    - Milestone JSON paths if hex ids still fail to match
  Ask the agent to diff tags and ids against 02 Pathway before you publish.

9. FOR THE AI AGENT (AFTER WORKSHEETS ARE FILLED)
================================================================================

AUDIENCE: Agent — interview, JSON generation, and handoff prompts.

--------------------------------------------------------------------------------
9.1 Interview protocol (if the designer says “write the conversations”)
--------------------------------------------------------------------------------
  Ask in this order:
    1) Mod technical Name and simGameConversations Manifest path
    2) List of scenes: design name + milestone beat id (ms_fp_*_Talk…)
    3) CastDef / Speaker id per scene (Darius vs Locals vs custom)
    4) Branch tags (every fp_choice_* to set and which milestones read them)
    5) Cancel / abort conversation needed? (TalkForCancel beat id)
    6) ExtendedConversations in load order? (Y/N — affects actions/conditions in JSON)
    7) Closest vanilla/mod .bytes to clone as JSON template (path or name)
    8) Placeholder ok until playtest? (9caaf9ff484fe7d2823c5d7e per 01 Reference)

  Stop if the designer wants tactical barks only — redirect to 04 Part 6.6.

--------------------------------------------------------------------------------
9.2 Generation order (recommended)
--------------------------------------------------------------------------------
  1) Complete §4 staging graph per scene → designer sign-off on branches/tags
  2) Obtain or export a CT Conversation JSON template (vanilla/mod match)
  3) Produce conv_fp_*.json per scene (§5 rules) + change log
  4) List deliverables: json path, intended .bytes stem, milestone beat ids affected
  5) Wait for designer CT import + hex ids (§6) before patching milestone JSON
  6) Patch ms_fp_* System_StartConversation "value" only after §6 hex ids confirmed
  7) Cross-check 02 Pathway: conversation branch + tag gates on branch beats
  8) Remind designer: 04 Part 9 checklist + in-game tag verification (§8.2)

--------------------------------------------------------------------------------
9.3 Prompts you should give the designer after delivering JSON
--------------------------------------------------------------------------------
  - "Please import conv_fp_*.json in ConverseTek (working directory set) and confirm
     every branch from 04b §2 appears in the editor. Reply if import fails or graph is empty."
  - "After save, paste the graph hex id for each scene into 04b §6 and 04 Part 6.4."
  - "Copy the final .bytes into Mods\<Name>\simGameConversations\ and confirm Manifest."
  - "Launch once and paste output_log lines containing StartConversation, MissingDef,
     or your mod Name."
  - "Play through each choice and confirm milestone Required Tags match §3 before
     the next contract or talk beat."
  - "If you edited JSON in CT and re-exported, send the updated .json so staging
     matches shipped .bytes."

--------------------------------------------------------------------------------
9.4 Stop conditions
--------------------------------------------------------------------------------
  - Do not patch milestone hex ids with guesses or .bytes filenames.
  - Do not invent Conversation JSON schema fields — require a CT export sample.
  - If import fails, request designer CT version + a known-good export JSON to diff.
  - If scene needs >10 responses per node, stop and redesign branches with the designer.
  - If Flashpoint actions require ExtendedConversations but mod stack lacks it, flag
    before delivery.
